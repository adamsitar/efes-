# LETVIS FDP — Build & Debug Environment

This repository contains the source of LETVIS FDP, a Slovak air-traffic-control system written in Borland Pascal 7 between 1993 and 2002: four DOS programs — OPL, EDD, FDA, SERVER — sharing a library tree under `fdp.source/FDP/LIB/`. The tooling here compiles them with debug symbols and runs and debugs them under DOSBox-X. All configuration paths are relative, so the folder is self-contained and copies between machines (the eventual target is native Windows; no symlinks anywhere). `HISTORY.md` records every problem that had to be solved to make this work — read it before assuming something here is accidental.

DOSBox-X mounts four drives, defined in `dosbox-x.conf`: `C:` is the demo working directory (`fdp.demo/idd/` — data files, the binaries you run, `INST\` config sets), `D:` is Borland Pascal 7 (`BP7/installed/` — `BPC`, `BP`, `TD` and friends), `E:` is the source tree (`fdp.source/`), and `L:` is a short alias for the shared libraries (`fdp.source/FDP/LIB/`) that exists to keep DOS command lines under their ~126-character limit. `dosbox-x.map` is the keymapper file; it unbinds Ctrl+F9 from "kill emulator" because that is Borland's Run key — do not delete it, and know that a mapper file must always be *complete* (a partial one silently unbinds the entire keyboard).

## Working in this repo (humans)

**Start an interactive session** with `./utils/run/run-x.sh` (Linux) or `utils\run\run-x.bat` (Windows). You land at `C:\>`. The demo-day scripts all have Windows twins: `utils\open\bp.bat`, `utils\open\td.bat`, `utils\deploy.bat`, `utils\build-headless.bat`, `utils\kill-dosbox.bat`. One Windows caveat: `dosbox-x.map` was saved by an SDL1 build; if a Windows (SDL2) DOSBox-X ignores it and Ctrl+F9 quits the emulator again, unbind it once on that machine via Main menu → Mapper editor → select the shutdown event → Del → Save.

**Compile** from the host with `./utils/build-headless.sh EDD` (targets: `OPL`, `EDD`, `FDA`, `SERVER`; ~5 s, fully headless), or inside DOSBox with `E:\BUILD.BAT EDD EDD` (SERVER lives in folder `S`). Compiler output lands in `fdp.source/COMPILE.LOG`; the binary in `fdp.source/FDP/<folder>/APP/`. The flags that matter: `/V` embeds the TD symbol table in the EXE (`/GD` does *not* — it only writes a map file). All units compile with `D+,L+` (set in `LIB/FDP/COMP_EXE.H`), so every module is source-debuggable in the IDE. Note that BPC builds and IDE builds write to the *same* output path but differ: BPC embeds the `/V` symbol table (894 KB EXE, what standalone TD needs), while the IDE keeps debug info in its own memory and writes a lean 140 KB EXE.

**Run** from `C:\`: type `EDD`. To run a fresh build, deploy it first: `./utils/deploy.sh EDD` copies the EXE *and* OVR (a matched pair from one compile — mixing generations gives `Overlay manager error (-1)`); `./utils/deploy.sh EDD --shipped` restores the 1999 originals. The demo dir holds one program's config at a time (this tree is seeded for EDD; `copy C:\INST\OPL\*.* C:\` switches to OPL). Two seeding traps: it overwrites the deployed binaries with shipped ones (re-deploy after), and it plants both a stale 1998 `ERR.LOG` (delete before diagnosing: `./utils/logs.sh err --clear`) and a stale `C:\COMP.H` that poisons IDE compiles (rename it away — the current one is parked as `COMP.H.1999-seed`). EDD's `COMP.H` is set to `DEBUG` off + `DEMO` on: `DEBUG` on would bake the original author's machine paths (`\xe\e\`) into the overlay loader and the binary won't start. If demo data gets into a bad state: `git restore fdp.demo`.

**Debug — primary workflow: the Borland Pascal IDE.** `./utils/open/bp.sh EDD` opens `BP.EXE` with `EDD.PAS` loaded, cwd `C:\` so the program finds its data. The IDE runs in DPMI protected mode and keeps itself and the debug info in extended memory, so the real-mode program gets its full 640 KB heap — this is the only debugger here that survives EDD's complete startup, and almost certainly how the author worked. Keys: **F9** compile (same engine and output as BPC), **Ctrl+F8** toggle breakpoint, **Ctrl+F9** run, **F7/F8** step into/over, **Ctrl+F4** evaluate, **Alt+F5** view the program's screen, **Alt+X** quit. Its one-time configuration (directory lists mirroring `fdp.source/FDP/EDD/BPC.CFG`, integrated debugging on) is saved in `fdp.demo/idd/BP.TP` and auto-loads. If Make picks the wrong target, set Compile → Primary file to `E:\FDP\EDD\APP\EDD.PAS`. Verified working: breakpoint on `MainApp:= New(PEDDapp,Init)`, step over the entire init, live radar UI under the debugger.

**Breakpoints from a file.** Keep browsing the code in your normal editor; write breakpoints into `breakpoints.txt` (one per line: `E:\FDP\EDD\APP\EDD.PAS 36` or `L:\FDP\SECTOR.INC 64` — library includes work; optional trailing condition like `pocet > 10`, `#` comments), then with the IDE open run `./utils/bp-load.sh` — it programs the live session through the Debug → Add breakpoint dialog, ~10 s per entry, hands off the keyboard while it runs. Breakpoints must sit on executable lines. **After every F9 rebuild, run `./utils/deploy.sh EDD` on the host before Ctrl+F9**: the overlaid program loads `EDD.OVR` from `C:\` at runtime, and a stale one there dies with `Error 209: Overlay file read error`.

**Debug — secondary: standalone Turbo Debugger**, for quick symbol-table inspection and breakpoints near startup. `./utils/open/td.sh EDD` opens TD on the *deployed* binary; `./utils/debug-x.sh EDD` builds, deploys, and opens it in one step. Source paths are passed via `-sd`; the full keystroke reference is `TD-CHEATSHEET.md`. Hard limitation: TD and its ~326 KB symbol table share conventional memory with the program, so stepping through EDD's full init dies with runtime error 203 (heap overflow). That is physics, not configuration — use the IDE for whole-program debugging. TD's auxiliary tools (`TDUMP`, `TDSTRIP`, `TDMEM`, `TD286INS`, `TDREMOTE`…) are installed in `BP7/installed/`; `TDUMP C:\EDD.EXE` dissects the symbol table when in doubt. TDX refuses real-mode programs; TD286 fails machine detection under DOSBox-X; TD386's driver loads (`DEVICE D:\TDH386.SYS`) but still can't fit the table.

Searching this tree: always `grep -a` — the box-drawing characters in comment banners make GNU grep classify half the files as binary and silently skip them.

## Working in this repo (agents)

The `utils/` directory is your hands and eyes on DOS. Scripts locate the repo root themselves. One live session at a time; **always clean up sessions you start** (`./utils/kill-dosbox.sh`, which the user can also run when you leak one — it uses `pkill -x`, because `-f` would match your own launching shell and kill it).

- `./utils/dos.sh 'DIR C:\' 'TYPE C:\ERR.LOG'` — batch: run DOS commands headlessly (no window, dummy SDL driver), each command's stdout captured to `fdp.source/AGENT.LOG` and echoed back. Times out (default 60 s, `DOS_TIMEOUT` env) — a timeout usually means the program reached an interactive UI. Graphics programs print errors to the *screen*; batch capture will not see them.
- `./utils/dos-live.sh [CMD ...]` — start a background session with a real window for interactive/graphics work (e.g. `./utils/dos-live.sh EDD`, or a TD/BP command line). Forces `output=surface` (otherwise screenshots are stale GL buffers), `usescancodes=false` (otherwise injected keys garble), `autolock=false` (otherwise the first click captures the mouse and coordinates go relative).
- `./utils/screenshot.sh [out.png]` — capture the window by id (root-coordinate grabs are black under WSLg) and print the path; read the image. Surface mode makes screenshot pixels equal click coordinates.
- `./utils/keys.sh F9` / `./utils/keys.sh --type 'EDD' Return` — keystrokes via XTEST after acquiring *and verifying* X focus, with stray modifiers released first. Under WSLg, focus is stolen whenever the user touches another window; the script then refuses or warns rather than dropping keys silently — re-send after a screenshot. XSendEvent (`--window`) delivers only plain chars/Return, never F-keys or Alt-combos; don't regress to it.
- `./utils/click.sh X Y [button]` — mouse click at window-relative (= screenshot) coordinates. Pointer events need no focus, so when the keyboard is hostage you can still drive TD/BP/DOSBox through their menu bars. Move-then-click with a beat between (the script does this) — the emulated cursor lags warps.
- `./utils/logs.sh [err|compile|agent] [--clear]` — read `ERR.LOG` (mind stale 1998 lines; `--clear` before reproducing), `COMPILE.LOG`, `AGENT.LOG`.
- `./utils/bp-load.sh [file]` — program breakpoints into a running IDE session from a text file (`FILE LINE [condition]` per line, default `breakpoints.txt`). Building block for scripted traces: load breakpoints, run, screenshot at each stop, continue.
- `./utils/build-headless.sh T` / `./utils/deploy.sh T [--shipped]` / `./utils/debug-x.sh T` / `./utils/open/bp.sh T` / `./utils/open/td.sh T` — as in the human section; `open/*` are foreground launchers, use `dos-live.sh` with the same command lines when you need to drive them yourself.

Recipes that work: *diagnose a startup failure* — `logs.sh err --clear`, `dos-live.sh`, `keys.sh --type 'EDD' Return`, wait, `screenshot.sh`, read the screen, then `grep -arn "<message>" fdp.source/` to find the check (every halt message is a string literal). *Drive a TUI* — screenshot, compute coordinates, click/keys, screenshot again; never assume a keystroke landed without looking. *Verify a build runs* — deploy the pair, run live, screenshot after ~10 s: EDD's UI up means success; a `C:\>` prompt back means it halted (read the screen above it). Sleep generously around emulator startup (~5 s) and program init (~10–15 s); never send keys during IDE/TD load. The DOSBox-X console (`/tmp/dosbox-live.log`) shows emulator-level trouble — e.g. a hidden quit-confirm prompt means a hotkey collision froze the core (Ctrl+F9 is unbound in `dosbox-x.map` for exactly this).

Source-change policy learned the hard way: prefer fixing *data* (the 1999 demo data predates the 2002 parser strictness) and *build configuration* over program logic; keep code-generation flags identical when adding debug info (`D,L,Y` are pure symbol switches; `Q,R,S` change behavior); and remember Pascal comments don't nest — a comment mentioning a `$D+` directive in braces ends at its closing brace.
