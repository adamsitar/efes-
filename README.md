# LETVIS FDP — Build & Debug Environment

This repository contains the source for LETVIS FDP, a Slovak air-traffic-control system written in Borland Pascal 7 between 1993 and 2002 by Ales Kosice. The codebase consists of four DOS programs (OPL, EDD, FDA, SERVER) plus a large shared library tree under `fdp.source/FDP/LIB/`. The setup here lets all four programs be compiled with full debug symbols on a Linux host, using DOSBox-X to host the original BP7 toolchain non-interactively.

## How to use it

**Running DOSBox-X interactively.** From the repository root, run `dosbox-x -conf dosbox-x.conf`. The configuration mounts three drives — `C:` is the demo data directory (`fdp.demo/idd/`), `D:` is the Borland Pascal 7 install (`BP7/installed/`), and `E:` is the source tree (`fdp.source/`) — and drops you at the `C:\>` prompt. From there you can run `OPL.BAT` or `EDD.BAT` to launch the shipped demo binaries, or `BP` to enter the BP7 IDE, or `TD <name>.EXE` to load Turbo Debugger.

**Compiling.** From the host shell, run `./build-headless.sh <TARGET>` where `<TARGET>` is one of `OPL`, `EDD`, `FDA`, or `SERVER`. The script launches DOSBox-X headlessly, runs `BPC.EXE` on the selected program with full debug-symbol flags (`/V`, `/GD`, `/$Y+`), captures the compiler output to `fdp.source/COMPILE.LOG`, then exits. A DOSBox window flashes briefly during the build; no interaction is needed. Each program takes around 5 seconds to compile from clean and produces a `.EXE`, `.MAP`, and a fresh set of `.TPU` files next to the sources. The output binary lands at `fdp.source/FDP/<folder>/APP/<TARGET>.EXE` — note that the `SERVER` target lives under the `S/` folder (everything else has matching folder and program names).

**Running a freshly-built program.** The programs expect a working directory full of data files (`.INI`, `.CHR`, `.BGI`, the `INST/` subtree, etc.) which only the demo directory has. To test a fresh build, swap it into the demo dir: back up the shipped binary first (`cp fdp.demo/idd/<NAME>.EXE fdp.demo/idd/<NAME>.EXE.orig`), then copy yours over (`cp fdp.source/FDP/<folder>/APP/<NAME>.EXE fdp.demo/idd/`). Launch DOSBox-X interactively and run `<NAME>.BAT` from the `C:\>` prompt. To restore the original demo binary, copy back from `.orig`.

**Loading a program into Turbo Debugger.** Launch DOSBox-X interactively, then at the prompt navigate to where the `.EXE` lives — either `C:\` (if you swapped your build into the demo dir) or `E:\FDP\<folder>\APP\` (the build location) — and run `D:\TD.EXE <NAME>.EXE`. If debug symbols are working you'll land in a window showing Pascal source with the cursor at the program's `begin` block; if not, TD will show only a disassembly window and a "program has no symbol table" message. The basic keys are F7 to step into, F8 to step over, F9 to toggle a breakpoint, F4 to run-to-cursor, and Ctrl+F7 to add a watch expression. There are three TD variants on `D:\` — `TD.EXE` for real-mode programs, `TD286.EXE` for 286 protected mode, and `TDX.EXE` for DPMI. Try `TD` first; fall back to `TDX` if it complains about the memory model.

## How we got here

The starting state, on the morning we began, was a source snapshot that had never been built end-to-end on any machine other than the original developer's, which no longer exists. The developer himself was reachable but had only 30-year-old memory to work with, and his attempts to remember the build details had reduced to "the library paths weren't set, but I don't remember which ones." Adam had inherited the project, had DOSBox-X running, but every compile attempt died at the same line — `..\LIB\UNIT\BGIGR4.INC(162): Error 3: Unknown identifier _write_err_log` — and the demo binaries he had ran but contained no debug symbols, which made it impossible to step through the code in Turbo Debugger to understand it. The goal was simple in shape and turned out to be subtle in execution: get all four programs compiled clean with debug symbols, from a single host-side command, so the codebase could finally be source-level debugged.

The first hour and a half went into diagnosis. The error was not what it looked like. `_write_err_log` is defined in `ERR_LOG.INC`, which is textually `$I`-included into `IO.PAS`'s implementation section, which in standard Pascal scoping would make the symbol private to `IO`. `BGIGR4.INC` lives inside `BGIGRAPH.PAS` — a different unit — and so on the surface the call looked structurally broken. We built up a theory that the source had been modified since the developer last successfully built, and that `IO.PAS`'s interface section had originally exposed these names via forward declarations which had since been deleted. We patched in those forward declarations and the build advanced past `BGIGR4` only to hit a new error: `vytvor_novy_subor` undefined. A grep across the entire source tree returned nothing. A delegated agent confirmed: no definition anywhere. We started writing up a recommendation that the codebase was structurally incomplete and would need stubs reconstructed.

Adam opened `IO_NU.INC` directly and found the function staring at him — clearly defined, exactly where it should have been. The reason grep had missed it (and the agent had missed it, and I had missed it) was that `IO_NU.INC` contains DOS box-drawing characters (`\xDC` Ü) used as ASCII-art borders in comment blocks. GNU grep classifies any file containing such bytes as binary and silently omits it from `-r` recursive output unless you pass `-a` to force text mode. That methodology failure had cost a real iteration, and I saved it to memory so future sessions on pre-Unicode codebases would default to `grep -ar`.

With the grep blind spot lifted, the second iteration of the investigation found the actual cause in five minutes. `IO.PAS` does `{$I io_arch.h}`. There are **two** files named `IO_ARCH.H` in the source tree — a 330-byte stub in `fdp.source/FDP/LIB/FDP/` containing only seven `const` declarations, and the real 2015-byte file in `fdp.source/FDP/LIB/UNIT/` that declares `archive_err`, `_write_err_log`, `write_err_log`, `vytvor_novy_subor`, `_another_line_err_log`, and a dozen other procedures whose implementations live in IO's `$I`-included `.inc` files. The `/I` search path in `BPC.CFG` lists `..\LIB\FDP` before `..\LIB\UNIT`, so BPC was silently picking up the stub and never seeing the real declarations. Every "unknown identifier" error we'd been chasing was a symptom of that one shadowed include. The whole "missing forward declarations" theory had been wrong from the start; the source was complete, the include-path resolver was just picking the wrong file. The fix was a single `git rm`: remove the stub. OPL built clean in 5.4 seconds, all the way through.

From there the rest of the work was infrastructure. The harness was extended to take a target argument (`./build-headless.sh OPL|EDD|FDA|SERVER`) by parameterizing both the DOS-side `BUILD.BAT` and the shell wrapper. The other three programs each got a copy of OPL's `BPC.CFG` placed in their folder so the relative paths would resolve correctly from each program's perspective. All four programs then built clean on the first attempt, which was strong evidence that the `IO_ARCH.H` shadow had been the only real structural issue in the codebase.

One more rake to step on. The debug-symbol flag the original `COMPILE.BAT` used was `/GD`, which I had carried over into our `BUILD.BAT` assuming "GD" meant "Generate Debug." When Adam tried to load `EDD.EXE` in Turbo Debugger and got `program has no symbol table`, the assumption broke. I'd never read BPC's own help. A one-line dump of `BPC` with no arguments revealed that `/GD` actually means *Detailed map file* — it produces a `.MAP` but does nothing for source-level debug. The real flag is `/V` (*Debug information in EXE*). Without `/V`, debug info ends up in the per-unit `.TPU` files (since `$D+` and `$L+` are default-on in BP7) but never gets linked into the final binary. Adding `/V` to the build command grew `EDD.EXE` from 154 KB to 914 KB, and TD opened it with the breakpoint set at the entry point and the Pascal source visible. The methodology lesson was archived alongside the grep one: verify compiler flags against the tool's own help, don't infer them from script names or comments.

Two false starts (the forward-declaration patch, the `/GD` flag), one genuine methodology failure (grep skipping DOS files), and one elegantly small fix (deleting a 330-byte orphan stub) — and the codebase that had been un-buildable from a clean tree for an unknown number of years now compiles all four programs in under thirty seconds with full debug symbols. The seven git commits on `master` (run `git log --oneline` to see them) document each step end-to-end, so the next person to touch this repository can read both the working state and how it got there. The harness itself is small enough — one shell wrapper, one DOS batch file, one DOSBox-X config — that it's easy to extend if more programs ever need to be added.
