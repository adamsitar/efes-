# LETVIS FDP — Build & Debug Environment

This repository contains the source for LETVIS FDP, a Slovak air-traffic-control system written in Borland Pascal 7 between 1993 and 2002. It consists of four DOS programs — OPL, EDD, FDA, SERVER — sharing a library tree under `fdp.source/FDP/LIB/`. The tooling here compiles all four with full debug symbols and runs them under DOSBox-X, on Windows or Linux. All paths in the configuration are relative, so the folder is self-contained and can be copied between machines.

## Usage

**Starting DOSBox-X.** Run `RUN-X.bat` (Windows) or `./run-x.sh` (Linux). Both launch DOSBox-X with the local `dosbox-x.conf`, which mounts four drives — `C:` is the demo data directory (`fdp.demo/idd/`), `D:` is the Borland Pascal 7 install (`BP7/installed/`), `E:` is the source tree (`fdp.source/`), and `L:` is a short alias for the shared library sources (`fdp.source/FDP/LIB/`) — and leaves the prompt at `C:\>`.

**Compiling.** Inside DOSBox, run `E:\BUILD.BAT <FOLDER> <NAME>`, e.g. `E:\BUILD.BAT OPL OPL` or `E:\BUILD.BAT S SERVER` (the SERVER program lives in the `S/` folder; the other three folders match their program names). From a Linux host, `./build-headless.sh <TARGET>` does the same headlessly, with `<TARGET>` one of `OPL`, `EDD`, `FDA`, `SERVER`. Compiler output is captured to `fdp.source/COMPILE.LOG`; the binary lands at `fdp.source/FDP/<folder>/APP/<NAME>.EXE`. Each program compiles from clean in a few seconds.

**Running.** The programs need the data files that live in the demo directory, so they run from `C:\`. Type `OPL` or `EDD` at the prompt. To run a freshly compiled build, copy it in first — both the EXE *and* its overlay file, which form a matched pair from the same compile: `copy E:\FDP\EDD\APP\EDD.EXE C:\` and `copy E:\FDP\EDD\APP\EDD.OVR C:\` (an EXE running against an older OVR dies at startup with `Overlay manager error (-1)`). Note that the demo directory holds *one program's* configuration at a time; this tree is set up for EDD. To switch programs, seed the target's config set: `copy C:\INST\OPL\*.* C:\` for OPL, `copy C:\INST\IDD\*.* C:\` for EDD. With the wrong set loaded, a program exits at startup and `ERR.LOG` records why (EDD without its set: `There is not defined RDP sector`). Seeding also restores the shipped, symbol-less EXE and OVR, so re-copy the debug pair afterwards. If the demo data ever gets into a bad state, the cleanest reset is `git restore fdp.demo` on the host (which returns to the last committed state).

**Debugging.** From `C:\`, run `TD OPL.EXE` (fall back to `TD286 OPL.EXE` if TD complains about memory). With the debug build in place, the Module window opens on the program source with the cursor at the entry point. F2 toggles a breakpoint, F7/F8 step into/over, F9 runs, Alt+X quits. A one-time setup is needed to also see source when stepping into library units: Options → Path for source, enter `E:\FDP\OPL\APP;L:\UNIT;L:\OOP;L:\NET;L:\L4;L:\FDP;L:\FRM;L:\MNU;L:\ARCH`, then Options → Save options. The full keystroke and workflow reference is in [TD-CHEATSHEET.md](TD-CHEATSHEET.md). On Linux, `./debug-x.sh <TARGET>` compiles if needed, copies the EXE into the demo dir, and opens TD in one step.

## How the build was fixed

The source snapshot had never been compiled outside the original development machine, and every attempt failed with `Error 3: Unknown identifier _write_err_log` deep in the library tree. The cause was an include-file shadow: two files named `IO_ARCH.H` existed — a 330-byte stub in `LIB/FDP/` and the real 2 KB header in `LIB/UNIT/` declaring the error-logging interface. The `/I` search path in `BPC.CFG` listed `LIB\FDP` first, so the compiler silently picked up the stub and never saw the real declarations. Deleting the stub fixed the build; all four programs then compiled cleanly with no source changes.

Two further fixes concern debugging rather than compilation. The original build scripts used `/GD`, which despite appearances only writes a linker map file; the flag that embeds a Turbo Debugger symbol table in the EXE is `/V`, which grows each binary from roughly 150 KB to 900 KB and is what makes source-level debugging possible. And BP7 records source file paths in the debug info exactly as the search paths spell them, so the `BPC.CFG` files use absolute paths (`E:\FDP\<folder>\APP`, `L:\UNIT`, …) rather than relative ones — that way Turbo Debugger resolves the main program source from any working directory. The `L:` mount exists to keep those paths within DOS command-line length limits.

One practical note for searching this codebase: many files contain DOS box-drawing characters in comment banners, which GNU grep classifies as binary, silently omitting them from recursive searches. Use `grep -ar` to search the tree reliably.
