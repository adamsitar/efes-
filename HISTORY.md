# History: what it took to debug LETVIS FDP in 2026

This file records every problem encountered on the way to a working source-level debugging workflow for EDD, in the order they were hit, each with its cause and fix. The repository contains a Slovak air-traffic-control system written in Borland Pascal 7 between 1993 and 2002; the goal was to compile it with debug symbols, run it on its 1999 demo data under DOSBox-X, and step through Pascal source with a debugger. Every entry below was a hard blocker at the time it appeared.

## Getting it to compile and show source (earlier sessions)

**The build failed everywhere with `Error 3: Unknown identifier _write_err_log`.** Two files named `IO_ARCH.H` existed: a 330-byte stub in `LIB/FDP/` and the real 2 KB header in `LIB/UNIT/`. The `/I` include path in `BPC.CFG` listed `LIB\FDP` first, so the compiler silently picked the stub and never saw the declarations. Deleting the stub fixed the build for all four programs (OPL, EDD, FDA, SERVER) with no source changes. Lesson that recurs throughout this history: Borland include resolution is silent about shadowing, and this codebase leans heavily on `{$I}` textual includes.

**Binaries had no debug symbols despite `/GD`.** The original build scripts used `/GD`, which despite its name only writes a linker `.MAP` file. The flag that embeds a Turbo Debugger symbol table in the EXE is `/V` (it grows EDD from 126 KB to ~900 KB). `/$D+` and `/$L+` are compiler defaults; `/V` is what makes the debug info reach the EXE.

**TD showed disassembly instead of source.** BP7 records source paths in debug info exactly as the search paths spell them. With relative paths in `BPC.CFG`, TD running from `C:\` could not resolve them. Fix: absolute paths (`E:\FDP\EDD\APP`, `L:\UNIT`, â€¦) in every `BPC.CFG`, plus a `-sd` source-dir list when launching TD. The `L:` drive (mounted at `fdp.source/FDP/LIB`) exists purely to keep those path lists under DOS's ~126-character command-line limit.

**"Program has no symbol table" after seeding demo config.** The demo dir's `INST\` sets contain the author's shipped, symbol-less binaries, and both the seeding copy (`copy C:\INST\IDD\*.* C:\`) and the `OPL.BAT`/`EDD.BAT` launchers overwrite whatever EXE sits in `C:\`. Deploy the debug build *after* seeding, and never run those BATs afterwards.

**EDD exited with code 1; `ERR.LOG` said `There is not defined RDP sector`.** The demo dir holds one program's configuration at a time and shipped OPL-configured. EDD needs its set seeded from `INST\IDD`. A related trap: the seed plants a 1998-dated `ERR.LOG` whose stale lines (`ndb.IB not found`) mislead diagnosis â€” delete `ERR.LOG` before reproducing anything, then trust only freshly dated lines.

**"Overlay manager error (-1)" at startup.** Two independent causes, both real. First, the EXE and `.OVR` are a matched pair from one compile; mixing a fresh EXE with an older OVR fails exactly this way. Second, the `{$DEFINE DEBUG}` build bakes the original developer's machine layout into the program: via `LIB/FDP/PROGRAM.H`, `path_exe = '\xe\e\'` for EDD, and the overlay loader does `OvrInit('\xe\e\edd.ovr')` â€” a path that exists only on a 1990s dev machine. With `DEBUG` off, `path_exe = ''` and the overlay loads from the current directory.

## Making a self-built binary actually run (this session)

**With `DEBUG` off the binary still died at startup â€” silently.** Nothing on stdout, nothing in `ERR.LOG`. The breakthrough was watching the actual screen (the programs print failure reasons to the display): `.\LIST\aip-m.IB -> Invalid value of area altitude profile (S0).` The message comes from `RunError` in `LIB/FDP/SECTOR.INC`; the altitude decoder `decode_l4_vyss` (`LIB/L4/_L4_VYS.INC`) requires `S`-prefixed altitudes to have exactly four digits (`altS_ = 4` in `_L4_VYS.H`) and rejects the demo data's `S0`. The 2002 source tree is stricter than the 1999 shipped binary that tolerated this data. Fix: conform the data â€” six lines in `fdp.demo/idd/LIST/AIP-M.IB`, `ALT/S0-â€¦` â†’ `ALT/S0000-â€¦` (semantically identical: ground level). After that, the self-built EDD reached its full UI with live flight strips. No program source was changed to make it run.

**The runnable build had almost no debug info.** `LIB/FDP/COMP_EXE.H` gates compiler directives on `DEBUG`: the non-DEBUG branch compiles every unit with `D-,L-` (no line info, no local symbols). Only two modules (under `COMP_EXT.H`'s local `{$D+,F+,L+,Y+}`) showed in TD. First attempt: flip the non-DEBUG branch to `D+,L+` globally â€” pure symbol info, code bytes identical (verified: same code size), runtime checks `Q-,R-,S-` untouched so behavior matches.

**TD corrupted module names and couldn't find `EDD.PAS`.** With all ~150 units carrying debug info, the picker showed garbage entries (`)â‰ˆ8Ă±)K7Â·8|â€¦`), blank rows, and "Source file `Ă®-` not found" for the main module â€” with the corruption *moving between TD loads*. `TDUMP` (recovered later, see below) explained it: the symbol table's name pool was 109,828 bytes, past the 64 KB boundary; real-mode TD garbles every name whose pool offset exceeds 16 bits. Fix: stop carrying line info for everything. Global `D-,L-` restored in `COMP_EXE.H`, and a local `{$D+,L+}` added directly to `EDD.PAS` and `AP.PAS` (right after their `{$I comp.h}`). The EXE dropped to 451 KB, the name pool under 64 KB, and TD then auto-opened the Module window on `EDD.PAS` source at the entry point. Breakpoints on Pascal lines hit correctly. (A Pascal trap inside the fix: comment braces don't nest, so a comment that *mentions* `{$D+,L+}` terminates at the inner `}` â€” cost one broken build.)

**Stepping past init died with exit code 203 â€” heap overflow.** Real-mode TD, its ~326 KB symbol table, and the debuggee all share the same 640 KB of conventional memory. Standalone EDD gets 608 KB and needs roughly 450 KB of heap for its data load; under TD it gets ~150 KB and `New(PEDDapp,Init)` fails. Every escape route on the standalone-TD side was tried: `-sm`/`-ye`/`-y` memory flags (symbols stay in conventional RAM); TDX, the DPMI-hosted debugger (refuses real-mode programs: "Not a protected mode program"); TD386, the 386 virtual host (its driver `TDH386.SYS` *does* load under DOSBox-X via the runtime `DEVICE` command â€” an earlier session's claim that TD386 "doesn't apply under DOSBox-X" was wrong â€” but it still reports "Not enough memory to load symbol table"); TD286, the 286 extended-memory host (its Phar Lap kernel crashes DOSBox-X's pentium core outright with `E_Exit: JMP Illegal descriptor type 1`, and under `cputype=286` fails machine detection: "BIOS checksum does not match"); TDINST (its "use expanded memory" option was already on; TD 3.2 has no symbols-in-EMS setting). Along the way the BP7 install disk images in `BP7/extracted/` yielded `TDUTILS.ZIP` with the entire missing toolbox â€” `TDINST`, `TD286INS`, `TDREMOTE`, `TDRF`, `TDSTRIP`, `TDUMP`, `TDMEM` â€” now unpacked into `BP7/installed/`.

**The actual solution: the Borland Pascal IDE (`BP.EXE`), which is how the original author almost certainly worked.** The IDE runs in DPMI protected mode (hence `RTM.EXE`, `DPMI16BI.OVL` in the install) and keeps itself *and* all debug information in extended memory â€” it reported ~14 MB free after compiling â€” while the real-mode debuggee keeps essentially the whole 640 KB. Same compiler engine as `BPC.EXE`, one F9 to build all 183k lines in ~5 s, integrated source-level stepping. Verified end-to-end: breakpoint on `MainApp:= New(PEDDapp,Init)` hits; F8 steps *over the entire application init* â€” the exact operation that killed TD â€” and lands on the next line; another F8 runs the live radar UI under the debugger.

Three problems stood between "IDE exists" and "IDE works", all now fixed:

1. **The IDE compiled with wrong conditional defines.** The IDE starts with cwd `C:\` (so the debuggee finds its data files), and Borland resolves `{$I comp.h}` from the *current directory before the include path*. `C:\` contained the author's 1999 `COMP.H` (defining `TRENAZER_UDP`, `LZ_RC` â€” a different installation!), which poisoned the build with `Error 3: Unknown identifier: SIM_UNIX`. Fix: renamed to `COMP.H.1999-seed`. Gotcha: re-seeding config from `INST\` copies a `COMP.H` back into `C:\` â€” re-rename after any seed.
2. **F9 with an include file in the active window compiles that include standalone** (bogus "Unknown identifier: bool" at its first line). Fix: Compile â†’ Primary file = `E:\FDP\EDD\APP\EDD.PAS`, or just keep `EDD.PAS` the active window.
3. **Ctrl+F9 â€” Borland's Run key â€” is DOSBox-X's "kill emulator" hotkey.** Pressing it froze the whole session on a hidden quit-confirm prompt written to the *host console*. Fix: a complete keymapper file (`dosbox-x.map`, wired via `mapperfile=` in `dosbox-x.conf`) with the `hand_shutdown` binding cleared. Critical detail: a *partial* mapper file replaces the entire binding set â€” a first attempt containing only the one unbind line disabled the whole keyboard. The full file must be dumped from DOSBox-X's Mapper editor (Main menu â†’ Mapper editor â†’ Save) and then edited.

The IDE's one-time configuration (Options â†’ Directories filled with the `BPC.CFG` lists, integrated debugging on) is saved as `C:\BP.TP` (`fdp.demo/idd/BP.TP`, tracked in git) and auto-loads because the IDE starts in `C:\`.

## Driving DOS programs agentically under WSLg (this session)

Building the `utils/` toolkit surfaced its own chain of emulator and display-server problems, each of which silently breaks automation if unknown:

- **Headless batch runs**: `SDL_VIDEODRIVER=dummy` runs DOSBox-X with no window at all; command output is captured by appending `>> E:\AGENT.LOG` to each DOS command (`utils/dos.sh`). Graphics-mode programs print failures to the screen, not stdout â€” batch capture misses them by design.
- **Screenshots came back black**: under WSLg's rootless XWayland the X root window is not composited, so grabbing root coordinates yields nothing. Capture the window surface by id instead: `ffmpeg -f x11grab -window_id <id>`.
- **Screenshots came back tiny and stale with OpenGL output**: `XGetImage` cannot read a GL-rendered window. Live agent sessions force `output=surface` (set by `dos-live.sh`), which also makes window pixels 1:1 with emulated pixels â€” click coordinates equal screenshot coordinates.
- **Synthetic keystrokes were garbled** (`EDD` arrived as `1)=88â€¦`): DOSBox-X's default raw-scancode translation mismatches XTEST/XSendEvent events. Fix: `usescancodes=false`.
- **F-keys and Alt-combos never arrived via `XSendEvent`** (plain characters and Return do). Only XTEST delivers the full keyboard â€” but XTEST follows X input focus.
- **WSLg yanks X focus** whenever the user touches any Windows window, and its WM implements no `_NET_ACTIVE_WINDOW` (xdotool `windowactivate` is a silent no-op; `getactivewindow` errors). `keys.sh` acquires focus with `windowfocus --sync`, *verifies* it, sends via XTEST, re-verifies, and warns loudly when keys may have been lost. While the user actively types elsewhere, keyboard injection is simply unavailable â€” by design it fails loud, not silent.
- **A latched modifier turns F3 into Shift+F3** (TD ignores it): an interrupted earlier injection can leave Shift/Ctrl logically held. `keys.sh` force-releases all modifiers first.
- **The first click "captured" the mouse** (DOSBox relative-motion mode), desynchronizing all subsequent coordinates: `autolock=false`. With it, pointer events deliver by position with no focus at all â€” mouse menus (`click.sh`) work even when the keyboard is hostage, which is how TD and the IDE were driven through their menu bars.
- **`pkill -f dosbox-x` killed the wrong thing** â€” the pattern matches any shell whose command line mentions dosbox-x, including the one launching the next instance. `kill-dosbox.sh` uses `pkill -x` (exact process name).
- **EDD runs mutate the demo tree**: archive rotation rewrites `LIST/*.IB` files and creates directories whose DOS names start with a high byte that maps to UTF-8 NBSP (`\xA0CMN`, `\xA0NET`). Runtime artifacts are gitignored; `git restore fdp.demo` resets the rest.

Three late finds while wiring up file-driven breakpoints (`utils/bp-load.sh`). First, **"Error 200: Division by zero" at startup, appearing out of nowhere**: the legendary Borland CRT-unit delay-calibration overflow on CPUs faster than ~200 MHz. With `cycles=max` the emulated speed drifts with host load, so every earlier run had simply stayed under the threshold by luck â€” `cycles=fixed 50000` (a fast 486, period-correct) makes it deterministic and gone. Second, **"Error 209: Overlay file read error" when running from the IDE**: the overlay RTL loads `EDD.OVR` from the *working directory* (`C:\`), so after every F9 rebuild the fresh OVR must be deployed there (`./utils/deploy.sh EDD`) or the program reads a stale, mismatched overlay. Third, the selective-debug-info restriction was **retired**: it existed only for real-mode TD's 16-bit name pool, and the IDE keeps debug info in extended memory â€” `COMP_EXE.H` now sets `D+,L+` globally, every module is breakpointable (verified by stopping inside `L:\FDP\SECTOR.INC` from a text-file breakpoint), and the IDE-built EXE is lean (140 KB, no embedded table) because symbols never touch the EXE at all. The instructions for reviving the selective mode live in the `COMP_EXE.H` comment should standalone TD ever be needed again.

One methodological lesson paid for repeatedly: **observe, don't theorize.** The silent startup failure, the TD name corruption, the frozen IDE, and the focus war were all solved the moment the actual screen, the actual symbol table (`TDUMP`), or the actual emulator log was looked at â€” and every error message in this codebase is a grep-able string literal away from its source check (`grep -a` always: the box-drawing comment banners make GNU grep treat half the tree as binary).
